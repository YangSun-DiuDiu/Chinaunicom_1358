## 第一章 建设背景

### 1.1 行业背景

随着我国城市化进程的不断推进，各类产业园区、科技园区、写字楼综合体、工业园区如雨后春笋般涌现。据不完全统计，全国各类产业园区已超过2万家，园区经济已成为区域经济发展的重要引擎。

在园区规模持续扩大的同时，园区管理面临着越来越多的挑战。传统的园区管理方式依赖人工巡检、纸质登记、电话沟通等手段，存在管理效率低下、信息传递滞后、数据无法追溯、应急响应不及时等突出问题。

### 1.2 政策驱动

国家发改委、工信部等部门近年来密集出台了一系列推进智慧园区建设的政策文件，明确提出要"推动产业园区数字化、智能化转型，建设智慧园区管理平台，提升园区运营管理效率和服务水平"。数字化转型已成为园区提质增效的必由之路。

### 1.3 项目目标

本方案旨在建设一套**智慧园区综合管理平台**，面向园区运营管理方，实现以下核心目标：

1. **设备统一监控**：对园区内网络设备、监控设备、录像设备等进行7×24小时实时监测，自动发现离线故障并生成维修工单；
2. **访客规范管理**：建立访客预约→审批→登记→离开的闭环管理流程，支持H5扫码自助登记；
3. **维修流程闭环**：设备故障自动告警→创建工单→转派分配→接收处理→维修确认的全流程管理；
4. **数据驱动决策**：通过驾驶舱数据大屏和工作量统计，为管理决策提供数据支撑。

## 第三章 痛点分析

### 3.1 传统园区管理的核心痛点

#### 痛点一：设备故障发现不及时，响应滞后

> **现状**: 园区内网络设备、监控摄像头、NVR录像机等24小时不间断运行，传统方式依赖人工定期巡检或用户报修才能发现故障。设备从发生故障到被发现的平均时间长达数小时甚至数天，影响园区正常运营。

> **解决**: 平台通过定时心跳检测（Ping），每分钟对所有设备进行在线状态探测。一旦发现设备离线，**30秒内自动创建维修工单并发送短信通知责任人**，将故障响应时间从"小时级"降至"分钟级"。

#### 痛点二：维修流程混乱，责任不清

> **现状**: 设备故障后通过电话、微信等方式口头通知，缺乏正式的工单记录。责任人不在或无法处理时，缺乏转派机制，导致故障长期无人跟进。维修完成后无确认环节，无法形成闭环。

> **解决**: 平台建立完整的**维修工单流转体系**——自动创建工单→短信通知→可转派他人→被转派人可接收或拒绝→维修完成后通过短信链接确认上报。每个环节都有记录可追溯。

#### 痛点三：访客管理依赖纸质登记，效率低下

> **现状**: 访客来访需要在前台手工填写纸质登记表，信息易丢失、难以检索、无法进行数据分析。访客预约与审批缺乏线上化手段，沟通成本高。

> **解决**: 平台提供**访客全流程线上管理**——预约→审批(含短信通知访客)→登记进入→扫码验证→记录离开，全流程电子化。同时提供H5扫码自助登记入口，访客到访前即可完成信息填写。

#### 痛点四：信息孤岛，缺乏统一数据视图

> **现状**: 设备状态、访客数据、维修记录分散在不同系统或Excel表格中，管理者无法一站式掌握园区整体运行状况。

> **解决**: 平台提供**双模式驾驶舱数据大屏**——设备驾驶舱实时展示在线率/告警/心跳数据，访客驾驶舱展示访客流量/审批状态/来访动态。所有数据一目了然。

#### 痛点五：维护人员工作量无法量化

> **现状**: 维护人员的维修工作靠口头分配，完成情况靠自觉，无法准确统计每个人的工作量，绩效考核缺乏数据支撑。

> **解决**: 平台提供**工作量统计页面**，按维护人员维度统计总工单数、已完成数、完成率等指标，为绩效考核提供客观数据。

### 3.2 痛点与功能对应表

| 痛点 | 严重程度 | 对应功能 | 改善效果 |
|------|----------|----------|----------|
| 设备故障发现不及时 | 🔴 严重 | 心跳检测 + 自动告警 | 响应时间从小时级降至分钟级 |
| 维修流程混乱 | 🔴 严重 | 维修工单流转系统 | 每起故障有记录、有跟进、有闭环 |
| 访客纸质登记 | 🟡 中等 | 访客全流程线上化 + H5自助 | 登记效率提升80% |
| 信息孤岛 | 🟡 中等 | 驾驶舱数据大屏 | 管理者一站式掌握全局 |
| 工作量难以量化 | 🟢 一般 | 工作量统计页面 | 数据驱动绩效考核 |
| 短信号码管理混乱 | 🟡 中等 | 阿里云短信统一配置 | 统一签名、模板管理、费用可控 |

## 第四章 平台功能详述

### 4.1 首页驾驶舱

首页是管理者掌握园区全局运行状况的统一入口，采用双模式设计，可一键切换"设备驾驶舱"与"访客驾驶舱"。

#### 4.1.1 设备驾驶舱

面向运维团队和管理层，实时掌握园区所有设备的运行状态。

**统计卡片区**

页面顶部展示四张核心统计卡片，数字带有count-to动态增长动画效果：

| 卡片 | 内容 | 数据来源 |
|------|------|----------|
| 设备总数 | 园区内已录入的所有设备数量 | `iot_device` 表 count |
| 在线设备 | 当前Ping可达的设备数量 | `iot_device` 表 status='ONLINE' |
| 离线设备 | 当前Ping不可达的设备数量（红色警示） | `iot_device` 表 status='OFFLINE' |
| 在线率 | 在线设备占总设备的百分比 | 实时计算 |

**【截图位置1】**: 设备驾驶舱页面全景截图，包含四张统计卡片、类型分布饼图、最新告警列表、最新心跳列表

**设备类型分布图**

ECharts饼图按设备类型（网络设备/监控设备/NVR/其他）展示各类设备的数量和占比，鼠标悬停显示详细数据。

**最新告警列表**

展示最近发生的设备离线告警记录，包含设备名称、IP地址、离线时间，支持点击"查看更多"跳转到故障预警页面。

**最新心跳列表**

展示最近的心跳检测记录，包含设备名称、Ping结果（成功绿色/失败红色）、检测时间。

#### 4.1.2 访客驾驶舱

面向安保团队和管理层，实时掌握访客动态和审批情况。

| 卡片 | 内容 |
|------|------|
| 今日访客 | 当日创建的预约总数 |
| 待审批 | 当前状态为PENDING的预约数 |
| 总访客 | 历史累计访客数量 |
| 今日进入 | 当日已登记的进入记录数 |

**【截图位置2】**: 访客驾驶舱页面全景截图，包含统计卡片、状态分布柱状图、最新来访列表

**预约状态分布图**

ECharts柱状图按预约状态（待审批/已通过/已拒绝/来访中/已完成）分类展示数量。

**最新来访列表 + 待审批列表**

分别展示最近来访记录和待审批预约，支持点击"查看更多"跳转到对应管理页面。

### 4.2 设备全生命周期管理

#### 4.2.1 设备列表

设备列表是设备管理的主页面，提供设备的增删改查、搜索筛选、导入导出等功能。

**设备字段**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 设备名称 | 文本 | 是 | 设备唯一标识，如"核心交换机-01" |
| 设备类型 | 下拉选择 | 是 | NETWORK(网络设备)/MONITOR(监控设备)/NVR(硬盘录像机)/OTHER(其他) |
| IP地址 | 文本 | 否 | IPv4格式，用于心跳Ping检测 |
| 型号 | 文本 | 否 | 设备型号，如"H3C S5500" |
| 位置 | 文本 | 否 | 设备安装位置，如"A栋3楼弱电间" |
| SNMP团体名 | 文本 | 否 | 默认public，用于SNMP信息采集 |
| SNMP端口 | 数字 | 否 | 默认161 |
| SNMP版本 | 下拉 | 否 | v2c(社区认证)/v3(用户认证) |
| 巡检周期 | 下拉 | 是 | 1-10分钟，心跳检测间隔 |
| 负责人 | 文本 | 否 | 故障时短信通知的联系人 |
| 联系电话 | 文本 | 否 | 接收短信的手机号码 |

**搜索筛选**

| 筛选项 | 类型 | 说明 |
|--------|------|------|
| 设备名称 | 文本输入 | 模糊匹配 |

| 状态 | 下拉选择 | ONLINE(绿色)/OFFLINE(红色)/UNKNOWN(灰色) |

| 查看端口 | 无特别权限 | 弹窗查看设备端口列表 |
| 列显隐 | - | 右侧工具栏自定义表格列显示 |

#### 4.2.5 SNMP管理

通过SNMP协议（v2c/v3）实时采集设备运行信息。

| 功能 | 说明 |
|------|------|
| 设备选择器 | 下拉选择已配置SNMP的设备 |
| 基本信息 | 系统名称(sysName)、系统描述(sysDescr)、运行时间(sysUpTime) |
| 端口状态 | 端口索引(ifIndex)、名称(ifName)、类型(ifType，已内置274种类型映射)、状态(UP绿色/DOWN红色)、速率(ifSpeed) |

### 4.3 维修工单流转系统

维修工单系统是本平台的核心创新功能，实现设备故障从"发现→派单→处理→确认"的完整闭环。

#### 4.3.1 工单状态流转

```
                    ┌──────────┐
   设备离线/手动报修  │  PENDING  │  待处理（初始状态）
          ──────────▶│  待处理   │
                    └────┬─────┘
                         │ 转派
                         ▼
                    ┌──────────┐
                    │ ASSIGNED │  已派发（等待被转派人确认）
                    │  已派发   │
                    └────┬─────┘
                         │
              ┌──────────┼──────────┐
              │ 接收      │          │ 拒绝
              ▼           │          ▼
       ┌──────────┐       │   ┌──────────┐
       │ ACCEPTED │       │   │ REJECTED │  退回原责任人
       │  已接收   │       │   │  已拒绝   │
       └────┬─────┘       │   └──────────┘
            │              │
            │ 维修完成      │
            ▼              │
       ┌──────────┐        │
       │COMPLETED │        │
       │  已完成   │◀───────┘
       └──────────┘
```

#### 4.3.2 自动创建机制

| 触发场景 | 流程 |
|----------|------|
| **心跳检测发现离线** | Ping超时 → 状态变更为OFFLINE → 自动创建维修工单(PENDING) → 生成确认token → 发送含确认链接的短信 |
| **管理员手动报修** | 设备列表页点击"报修" → 创建维修工单(PENDING) → 生成确认token → 发送含确认链接的短信 |

#### 4.3.3 转派功能

当当前责任人因故无法处理时，可将工单转派给其他维护人员：

| 操作 | 执行人 | 效果 |
|------|--------|------|
| **发起转派** | 当前责任人 | 填写被转派人姓名、电话、转派原因 → 工单状态变为ASSIGNED → 被转派人收到短信通知 |
| **接收转派** | 被转派人 | 确认接收 → 工单状态变为ACCEPTED → 责任转移至被转派人 |
| **拒绝转派** | 被转派人 | 填写拒绝原因 → 工单状态变为REJECTED → 退回到原责任人 |

#### 4.3.4 短信维修确认

责任人收到的短信内容示例：


#### 4.6.2 角色管理与权限控制

系统采用四层权限控制模型：

| 层次 | 实现方式 | 效果 |
|------|----------|------|
| **路由控制** | 前端路由 `meta.permissions` 过滤 | 无权限的菜单不显示、页面不可访问 |
| **按钮控制** | `v-hasPermi` 指令 | 无权限的按钮从DOM中移除，不可见不可操作 |
| **API控制** | `@PreAuthorize("@ss.hasPermi(...)")` | 后端接口层第二道防线，防止绕过前端直接调用API |
| **数据权限** | `@DataScope` 注解 | 按角色数据范围（全部/自定义/本部门/本部门及以下/仅本人）过滤查询结果 |

**角色权限配置**：
- 菜单权限：树形勾选，可精确到按钮级别
- 数据权限：全部数据/自定义部门/本部门/本部门及以下/仅本人数据
