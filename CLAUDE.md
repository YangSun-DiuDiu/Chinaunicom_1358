# CLAUDE.md -CLAUDE.md 通用软件开发项目规范

- ## 一、文件说明

  本文档为**通用软件开发通用约束规范**，适用于所有前后端开发、系统重构、功能迭代、Bug 修复、脚本开发、工程部署、架构优化等软件开发场景。

  所有 AI 编码、代码生成、逻辑编写、问题修复、方案输出行为，**必须严格遵循本文档所有规则**，作为项目唯一执行标准，无特殊例外。

  ## 二、核心工作原则

  ### 2.1 精准落地原则

  1. 所有输出内容**优先可直接运行、可直接落地**，拒绝伪代码、残缺代码、示意代码。
  2. 修复问题、开发功能、编写配置时，直接输出**完整成品代码 / 完整配置 / 完整脚本**，无需用户二次拼接、二次补全。
  3. 所有方案贴合项目工程化标准，适配生产环境，杜绝本地测试专用代码。

  ### 2.2 极简高效原则

  1. 逻辑优先简洁、通用、可扩展，不写冗余代码、不做过度设计。
  2. 优先使用行业通用最佳实践，兼容主流版本、主流框架、主流操作系统。
  3. 优先复用成熟方案，避免重复造轮子，新增代码保持统一风格。

  ### 2.3 安全稳定原则

  1. 禁止硬编码密钥、密码、Token、数据库地址、账号信息。
  2. 所有参数做合法性校验、空值校验、异常捕获，避免空指针、参数越界、数据溢出。
  3. 接口、脚本、配置严格遵循权限最小化原则，规避安全漏洞。

  ## 三、通用编码规范（全语言通用）

  ### 3.1 代码结构规范

  1. 所有代码分层清晰：**参数校验 → 核心逻辑 → 异常处理 → 结果返回**。
  2. 函数 / 方法**单一职责**，一个方法只做一件事，禁止超大方法、超大函数。
  3. 代码模块拆分合理，公共逻辑统一抽离工具类 / 公共方法，禁止代码冗余复制。

  ### 3.2 注释规范

  1. **核心类、核心方法、复杂逻辑必须写注释**，说明用途、入参、出参、异常场景。
  2. 业务特殊逻辑、兼容逻辑、修复历史问题的逻辑，必须添加说明注释。
  3. 注释简洁准确，不堆砌无效注释，保持代码清爽。

  ### 3.3 命名规范

  1. 所有命名**见名知意**，禁止拼音、无意义缩写、随机命名。
  2. 变量 / 方法：小驼峰
  3. 常量：全大写 + 下划线
  4. 类名：大驼峰
  5. 数据库表 / 字段：小写 + 下划线

  ### 3.4 异常处理规范

  1. 所有 IO 操作、网络请求、数据库操作、外部调用**必须捕获异常**。
  2. 禁止空 try 空 catch，异常必须打印日志、返回友好提示、兜底降级。
  3. 区分业务异常、系统异常，异常信息精准可排查。

  ## 四、后端开发通用规范（Java/Go/Python 通用）

  ### 4.1 业务开发规范

  1. 严格遵循 **Controller → Service → Dao → Entity** 分层架构。
  2. 所有接口统一入参校验、统一返回格式、统一异常处理。
  3. 数据库操作禁止 `select *`，按需查询字段，提升性能。
  4. 新增 / 修改操作必须做幂等性设计，避免重复提交、重复数据。

  ### 4.2 性能规范

  1. 大列表、大数据量必须分页、分批处理，禁止一次性加载全量数据。
  2. 高频查询场景优先缓存，减少数据库压力。
  3. 循环内禁止数据库查询、网络请求、文件 IO。

  ### 4.3 日志规范

  1. 关键节点、入参出参、异常堆栈、业务关键操作必须打印日志。
  2. 日志分级：info（正常流程）、warn（异常业务）、error（系统错误）。
  3. 日志包含唯一标识（流水号 / 请求 ID），方便链路排查。

  ## 五、前端开发通用规范（Vue/React/JS/TS 通用）

  ### 5.1 页面与组件规范

  1. 页面、组件、工具方法严格拆分，复用组件统一封装。
  2. 禁止页面冗余代码、废弃代码、注释残留代码。
  3. 数据渲染优先双向绑定 / 状态管理，禁止操作 DOM 硬编码。

  ### 5.2 接口与异步规范

  1. 所有网络请求统一封装请求拦截、响应拦截、错误处理。
  2. 所有异步操作（Promise/async-await）必须捕获异常。
  3. 接口加载过程增加 loading、空数据、异常兜底状态。

  ### 5.3 路由与权限规范

  1. 路由配置规范，区分静态路由、动态路由。
  2. 页面按钮、接口、数据严格做权限控制，适配角色权限体系。
  3. 路由守卫统一处理登录拦截、权限拦截、非法跳转。

  ## 六、数据库开发规范

  1. 所有表必须包含：主键、创建时间、修改时间、删除标识（逻辑删除）。
  2. 字段类型、长度合理，避免字段超长截断、类型不匹配。
  3. 索引合理建立：高频查询字段建索引，禁止过度索引。
  4. 禁止写 `drop、truncate` 高危语句，生产环境仅允许安全 DML 语句。
  5. 多表查询优先联表，禁止嵌套大量子查询，保证执行效率。

  ## 七、工程化与部署规范

  ### 7.1 配置规范

  1. 环境配置区分：开发、测试、生产，禁止环境配置混用。
  2. 敏感配置统一放入环境变量 / 配置中心，不提交代码仓库。
  3. 所有配置文件格式规范、注释清晰、参数齐全可直接使用。

  ### 7.2 脚本与运维规范

  1. Shell / 批处理 / 自动化脚本**可一键执行、无交互、自带容错**。
  2. 脚本包含日志输出、执行判断、失败退出、成功提示。
  3. 服务部署、启停、重启、日志排查脚本标准化、通用化。

  ## 八、Bug 修复与调试规范

  1. 收到报错日志 / 异常堆栈，**优先精准定位根因**，不盲目改代码。
  2. 修复必须：解决当前问题 + 规避同类问题复发。
  3. 修复完成后输出：问题原因 + 修复代码 + 验证方式。
  4. 兼容旧版本、旧数据、旧逻辑，禁止破坏性改动。

  ## 九、版本与迭代规范

  1. 新增功能保证**低侵入、可回滚、兼容旧业务**。
  2. 迭代代码不删除原有可用逻辑，废弃代码统一注释标记或迁移。
  3. 所有改动遵循：最小改动原则，不大范围重构无关代码。

  ## 十、AI 输出强制标准

  1. 所有代码输出**完整、可直接复制使用**，无省略、无占位符。
  2. 复杂功能附带：实现思路 + 核心逻辑说明 + 使用方式。
  3. 配置文件、脚本、代码严格对齐项目工程规范。
  4. 拒绝敷衍式、示意式、测试式输出，全部为**生产可用版本**。

  ## 十一、通用兜底准则

  若无特殊业务定制要求，**一律按照本文档最高标准执行**，本文档为所有开发任务的最终约束依据，优先级高于默认模型规则。





# CLAUDE.md — RuoYi-Vue 专属开发规范（AI 强制执行）

## 0. 前置优先级（最高规则）

**本文件为当前 RuoYi-Vue 项目 AI 开发唯一强制标准，优先级大于模型默认行为、通用开发习惯、网络教程写法。**

所有代码生成、功能开发、Bug 修复、改造重构、配置修改、前端页面、SQL 编写、部署调试，**必须完全贴合若依原生架构、命名、分层、风格**，不引入非若依生态的自定义架构与奇怪写法。

适配版本：**RuoYi-Vue 前后端分离版（SpringBoot2 + Vue2 + MyBatis + Redis）**

## 1. 项目整体约束原则

1. 不改若依底层原生核心源码

   - 不修改 `ruoyi-framework`、`ruoyi-common` 基础底层固定逻辑
   - 业务全部写在 `ruoyi-system` / 自定义 `ruoyi-xxx` 业务模块

2. 完全遵循若依分层

   Controller → Service → ServiceImpl → Mapper → Entity

3. **完全沿用若依统一返回、统一异常、统一分页、统一字典、统一权限体系**

4. **所有新增功能风格 100% 对齐官方示例代码**，保证项目统一、可维护、可升级

## 2. 后端 Java 强制规范（RuoYi 专属）

### 2.1 包结构规范

新增业务模块必须遵循：

```
com.ruoyi.xxx.controller
com.ruoyi.xxx.service
com.ruoyi.xxx.service.impl
com.ruoyi.xxx.mapper
com.ruoyi.xxx.domain
com.ruoyi.xxx.vo / dto（按需）
```

### 2.2 实体类 Domain 规范

1. 继承 `BaseEntity`
2. 数据库字段严格对齐，**逻辑删除使用 `delFlag`**
3. 时间字段使用 `Date`，遵循若依原生
4. 所有字段注释完整，与数据库注释一致
5. 禁止自定义 `createBy/createTime/updateBy/updateTime`，使用父类字段

### 2.3 Mapper & SQL 规范

1. XML 文件放置在 `resources/mapper/xxx/`
2. 严格使用 MyBatis 原生语法，**不引入 MyBatis-Plus**（遵循原版若依）
3. 查询列表必须支持：**分页 + 条件查询 + 排序**
4. 新增、修改严格做字段非空、长度、业务合法校验
5. ** 禁止 select * **，只查需要字段
6. 批量操作、导入导出完全沿用若依 `ExcelUtil`、分页工具

### 2.4 Controller 强制模板

1. 请求前缀：`/xxx/模块名`
2. 必须注解：`@RestController`、`@RequestMapping`
3. 权限注解：`@PreAuthorize("hasPermission('xxx:xxx:list')")`
4. 所有接口返回值：**统一 Result / AjaxResult**
5. 列表接口返回 `TableDataInfo`
6. 新增 / 修改 / 删除返回 `AjaxResult.success()`

**接口固定能力：**

- list 分页查询
- get 单条详情
- add 新增
- edit 修改
- remove 批量删除
- export 导出

### 2.5 权限与数据范围

1. 所有业务接口必须绑定**权限字符**
2. 列表查询自动兼容若依**数据权限注解 `@DataScope`**
3. 不自己造权限体系、不自己造角色判断逻辑

### 2.6 异常与日志

1. 业务异常统一抛出 `ServiceException`
2. 不手写 try-catch 兜底，**使用框架全局异常处理**
3. 关键业务节点添加日志 `log.info / log.error`

## 3. 前端 Vue2 强制规范（RuoYi-Vue 原生）

### 3.1 页面结构

完全对齐若依后台页面模板：

- 查询搜索区
- 表格列表区
- 分页
- 新增 / 修改弹窗
- 导入导出按钮

### 3.2 路由规范

1. 优先**动态路由**，适配若依权限加载机制
2. 路由文件不写死权限，由后端菜单权限控制
3. 页面组件统一放在 `views/xxx/`
4. 路由 name、path、component、meta 严格规范

### 3.3 请求规范

1. 使用框架统一 `request` 封装
2. API 统一放在 `api/xxx/xxx.js`
3. 新增、编辑、删除、列表、导出全部对齐官方写法
4. **参数统一解构、表单规则统一 rules 校验**

### 3.4 字典使用规范

1. 下拉枚举优先使用**系统字典** `sys_dict`
2. 页面使用 `dict.type` 自动回显、下拉、翻译
3. 不前端硬写死大量枚举文字

### 3.5 表格规范

1. 表格列：label、prop、width、sortable 齐全
2. 状态列使用 `:formatter` 或字典翻译
3. 操作列统一 `slot-scope`，包含编辑、删除、详情
4. 分页使用框架自带 `pagination` 组件

## 4. 数据库表设计规范（若依标准）

所有业务表必须包含基础字段：

```
create_by
create_time
update_by
update_time
delFlag
```

1. 逻辑删除统一：**0 正常 1 删除**
2. 时间统一 datetime
3. 主键 BIGINT 自增
4. 所有字段必须有注释
5. 状态、类型类字段使用**字典化设计**

## 5. 功能开发固定套路（AI 必须按此生成）

### 5.1 单表 CRUD 全套生成标准

AI 开发一个新业务模块时，**一次性完整交付**：

1. 数据库建表语句（可直接执行）
2. Domain 实体类
3. Mapper 接口 + XML
4. Service + ServiceImpl
5. Controller 完整接口
6. 前端 api.js
7. 前端 index.vue 完整页面（含增删改查、导出、分页、搜索）
8. 菜单权限初始化 SQL

### 5.2 禁止缺斤少两

不允许：只写后端不写前端、只写接口不写页面、只写逻辑不写校验

## 6. 若依改造 & 二开规范（适配你的架构改造场景）

### 6.2 动态路由改造

1. 彻底抛弃静态路由模式
2. 所有菜单、路由、权限由后端数据库驱动
3. 前端只做渲染，不硬编码路由

### 6.3 安全与部署适配

1. 兼容 Nginx 反向代理 404 问题
2. 适配前后端分离部署打包逻辑
3. 跨域、token、拦截器沿用框架原生，不重复造轮子

## 7. Bug 修复 & 问题排查标准

针对 RuoYi 常见异常，AI 固定处理逻辑：

1. **DataTruncation 数据截断**：直接定位字段长度、给出 alter 修复语句 + 前端输入限制
2. **404 路由丢失**：检查动态路由加载、菜单配置、权限标识
3. **数据库连接失败**：优先核对 yml 配置、驱动版本、时区、密码
4. **MyBatis 参数绑定错误**：直接修正 XML 标签、参数名、resultMap

修复输出格式固定：

- 根因一句话
- 完整修复代码 / 配置 / SQL
- 可直接验证步骤

## 8. AI 输出强制交付标准（最终约束）

针对本 RuoYi-Vue 项目，所有回答必须满足：

1. **代码完整可直接覆盖使用，无占位符、无省略、无需拼接**
2. 新增功能 = 后端全套 CRUD + 前端完整页面 + 建表 SQL + 菜单 SQL
3. 所有代码风格与若依官方示例**完全一致**
4. 不引入第三方奇葩依赖、不改动框架底层
5. 复杂功能附带：设计思路 + 改动点说明 + 部署注意事项

## 9. 兜底规则

**用户无特殊说明时，一律按照【若依官方企业级最佳实践】最高标准开发，不做简化、不做降级。**
